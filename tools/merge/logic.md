这份代码的核心业务是一套**安全且保守的“定向视频搬运与清理”机制**。

它的主要应用场景是：将临时目录（Input）中处理好的视频，归档到最终的媒体库目录（Output）中，并在确认归档完成后，安全地清理掉临时目录释放空间。

以下是该程序的详细业务逻辑拆解：

### 一、 核心运行模式
程序采用**双模式**运行设计，防止误操作：
1. **预览模式（默认）**：只读不写。扫描并打印出所有将要执行的动作（发现哪些匹配、计划跳过哪些、计划复制哪些、计划删除哪些），但不改变任何磁盘数据。
2. **执行模式（带有 `--apply` 参数）**：真正执行磁盘的复制和删除操作。

---

### 二、 业务执行流程

#### 1. 目录扫描与匹配逻辑（寻找目标）
* **不深入底层**：程序只扫描 `Input` 和 `Output` 目录下的**第一层子文件夹**（不递归深层目录，也不扫描目录外的散落文件）。
* **精准匹配**：提取两边文件夹的名称，找出**名称完全一致**的文件夹作为“待处理对象”。名称不一致的文件夹直接忽略。

#### 2. 文件过滤逻辑（识别目标）
* 在找出的“匹配文件夹”中，程序只关注**视频文件**（通过判断后缀名是否属于预设的 `.mp4`, `.mkv`, `.iso` 等集合）。
* 如果该文件夹内没有任何视频文件，程序直接跳过该文件夹。

#### 3. 搬运逻辑（安全移动机制）
程序遍历 `Input` 文件夹中的每一个视频，执行以下判断：
* **场景 A：目标端已存在同名视频**
  * **动作**：跳过复制操作（不覆盖、不产生额外的磁盘读写）。
  * **后续**：打上“源文件可删除”的标记。
* **场景 B：目标端没有同名视频**
  * **动作**：执行完整的 `Copy` 操作，将视频从 Input 复制到 Output 的对应文件夹中。
  * **验证**：如果复制过程中发生任何异常（如磁盘满、无权限），捕获错误并保留源文件。
  * **后续**：如果复制成功，打上“源文件可删除”的标记。

#### 4. 碎片清理逻辑（删除单个视频）
* 只要上述步骤中，视频被成功打上了“源文件可删除”的标记（即确认目标端已经有这份数据了），程序就会立即删除 `Input` 端对应的源视频文件。
* *（注意：此处使用 `Copy + 确认 + 删除源文件` 来代替系统级别的 `Move`，以此确保在异常断电等极端情况下，数据绝不丢失。）*

#### 5. 文件夹销毁逻辑（安全回收空文件夹）
当一个文件夹内的**所有视频**都被成功处理（要么成功搬运，要么目标端已存在）后，程序开始评估是否可以删除 `Input` 端的这个源文件夹：
* **安全检查**：检查该文件夹内是否还有残留文件。
* **忽略系统垃圾**：如果发现残留文件仅仅是 macOS 的 `.DS_Store` 或 Windows 的 `Thumbs.db`、`desktop.ini`，程序视其为“空文件夹”。
* **执行删除**：
  * **条件满足（完全空）**：彻底删除该文件夹，释放空间。
  * **条件不满足（有非视频文件）**：如果里面还有字幕文件（`.srt`）、封面图（`.jpg`）或子文件夹，程序会**中止删除动作**，保留该文件夹并打印警告，将非视频数据的处理权交还给用户人工处理。

### 三、 逻辑总结图示

```text
开始
 ├─ 提取 Input 和 Output 第一层子文件夹
 ├─ 取交集，找出同名文件夹
 │
 ├─ 循环：处理每一个同名文件夹
 │   ├─ 扫描 Input 中的视频文件
 │   │
 │   ├─ 循环：处理每一个视频
 │   │   ├─ 目标是否存在？
 │   │   │   ├─ 是：跳过复制，删除 Input 源视频
 │   │   │   └─ 否：复制到 Output，成功后删除 Input 源视频
 │   │
 │   └─ 所有视频都处理成功了吗？
 │       ├─ 否：跳过文件夹删除，保留现场
 │       └─ 是：里面还有字幕/图片等非视频文件吗？
 │           ├─ 是：跳过文件夹删除，防止误删用户其他数据
 │           └─ 否 (或只有系统隐藏垃圾)：安全删除整个 Input 文件夹
 │
结束
```
